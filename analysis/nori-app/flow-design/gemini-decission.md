Nori Flow Architecture: The UI-Driven Engine
Date: 2026-03-03
Status: Approved foundational architecture
Key Insight: A software engineer makes difficult things easy; a developer makes things that work but are hard. To scale Nori across languages without massive cognitive load or fragile code-generation, we must remove orchestration from the AI's hands. The UI generates a JSON blueprint. A "dumb" generic engine reads the JSON. The AI only writes pure, isolated functions.

The Philosophy: The "Dumb" Architecture
We are treating Nori's flow system as an embedded Workflow Engine.

Historically, AI struggles to write reliable orchestrators across multiple languages because gluing steps together, handling fatal errors, and emitting events requires perfect, context-heavy syntax.

To solve this, we enforce a strict separation of concerns:

The Brain (UI + JSON): The human uses the Nori UI to define the flow visually. The UI generates a strict JSON schema. This JSON is the absolute source of truth.

The Muscle (AI + Code): The AI is restricted to writing tiny, single-purpose pure functions or isolated UI widgets.

The Nervous System (The Engine): A lightweight, generic runtime engine reads the JSON and executes the AI's functions.

By doing this, the AI never writes an if statement for routing, and never writes an emitter.emit() call.

1. Backend Flows: The Linear Pipeline
   Backend flows are strictly linear. One path forward.

The JSON Blueprint (Generated by UI)
The UI generates a simple array of steps, defining what runs and whether failures are fatal:

JSON
{
"type": "pipeline",
"steps": [
{ "id": "01-validate", "action_id": "validateInput", "fatal": true },
{ "id": "02-write-db", "action_id": "writeToDatabase", "fatal": true }
]
}
The AI's Job (Pure Functions)
The AI writes the validateInput action. It does not know it is part of a flow. It just takes input and returns a result.

TypeScript
// actions/validate.ts
export async function validateInput(input: any) {
if (!input.name) return { success: false, error: "Missing name" };
return { success: true, data: input };
}
The Engine (The 80-line for loop)
The project owns a tiny, generic engine (~80 lines). It loops over the JSON, calls the registered functions, handles context passing, and automatically emits events:

TypeScript
// Pseudo-code of the Dumb Engine
for (const step of json.steps) {
emit(`step:${step.id}:started`, ctx);
const result = await registry.execute(step.action_id, ctx);

    if (!result.success) {
        if (step.fatal) return result; // Engine handles flow control
        continue;
    }
    ctx = result.data; // Pass output to next step
}
2. Frontend Flows: The 2D State Machine
   Frontend UI flows are not linear. They are 2-dimensional graphs driven by user clicks, API responses, and SSE events.

The JSON Blueprint (Generated by UI)
The UI generates a state machine defining the nodes (states) and the arrows connecting them (transitions).

JSON
{
"type": "state-machine",
"states": {
"viewing": { "renders": "KnowledgeDetailWidget" },
"editing": { "renders": "KnowledgeFormWidget" }
},
"transitions": [
{ "from": "viewing", "to": "editing", "on": "user:edit_clicked" },
{ "from": "editing", "to": "viewing", "on": "user:cancel_clicked" }
]
}
The AI's Job (Dumb Widgets)
The AI writes KnowledgeFormWidget.tsx. It does not write routing logic or global state. It just renders a form and triggers a generic on_submit event.

The Engine (The 100-line Hook)
A generic useFlowMachine hook acts as the frontend engine. It reads the JSON, listens for the user:edit_clicked event, and automatically swaps the rendered component from KnowledgeDetailWidget to KnowledgeFormWidget.

What is NOT a Flow
To prevent over-engineering, we strictly define what bypasses this system:

Navigating pages (Standard router)

Opening sidebars/modals (Local component state)

Simple data tables (Standard fetch + render)
Flows are explicitly reserved for multi-step operations with side effects (e.g., Wizards, Sync processes, Complex forms).

3. The Migration Path
   We will not rewrite the world on day one. We will bridge the gap in two phases:

Phase 1: The Linter (Option 1)
While we build the visual UI and the runtime engines, we continue to let the AI write the manual orchestrator glue code. However, we introduce a strict flow-lint script that ensures the AI's code matches the JSON blueprint. This stops the immediate bleeding of AI hallucinations.

Phase 2: The Runtime (Option 3)
Once the UI can reliably output the JSON schema, we delete the AI-generated orchestrators. We drop in the 80-line backend runner and the 100-line frontend hook. The JSON becomes the literal execution plan. The AI is permanently relegated to writing isolated workers.
